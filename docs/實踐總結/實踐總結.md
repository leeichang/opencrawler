## 2. 實際驗證的設計決策(順利)

### 2.1 Avalonia 11 + SqlSugarCore 5.x + SQLite

- Avalonia 原生控制項庫對此類 CRUD + 樹狀 + TabControl 的需求完全足夠,沒有被迫去找第三方套件。
- SqlSugar 的 CodeFirst 搭配 SQLite 免寫 migration SQL,開發期間加欄位只要在 Entity 上加屬性即可。
- 測試資料夾全用 `Path.GetTempPath() + Guid` 方式隔離,每個測試自己的 SQLite 檔、自己的文章資料夾,不污染全域。

### 2.2 .NET 9 target + .NET 10 runtime

本機只裝了 .NET 8 / 10 runtime,沒有 .NET 9。利用 `<RollForward>LatestMajor</RollForward>` 讓 `net9.0` 組出來的二進位在 .NET 10 上跑。建置 SDK 是 .NET 10,`dotnet run` 自動 roll-forward,沒再花時間安裝 .NET 9 runtime。

### 2.3 Core / App / Cli 三分

把 `Core` 切成獨立 class library 不依賴 Avalonia 這件事,在 **寫測試** 時回饋最明顯:CLI 整合測試只需 reference Core,不會被 UI 相依性拖累;CLI 和 GUI 完全共用同一份下載邏輯,做到「一處改、兩邊生效」。

### 2.4 WireMock.Net

自架模擬站比依賴外網穩定太多。IT-02 同 URL 重複下載、IT-03 404 失敗案例都靠這個跑。

## 3. 踩坑紀錄(每個坑值一段)

### 坑 1. SqlSugar + SQLite 的 AUTOINCREMENT

**症狀**:第一次跑 CLI 時 `DbInitializer.Init` 丟出 `AUTOINCREMENT is only allowed on an INTEGER PRIMARY KEY`。

**根因**:SqlSugar 對 C# `long` 預設生成 `BIGINT`,但 SQLite 只認字面 `INTEGER`(且必須大寫)才能把欄位視為 ROWID alias、允許 AUTOINCREMENT。

**解法**:所有主鍵加 `ColumnDataType = "INTEGER"`。

```csharp
[SugarColumn(IsPrimaryKey = true, IsIdentity = true, ColumnDataType = "INTEGER")]
public long Id { get; set; }
```

教訓:**SqlSugar + SQLite 必須手動指定 PK 型別字串**,不能依賴它的預設對映。

### 坑 2. 測試第二跑時「資料庫被占用」

**症狀**:第一個測試 pass,第二個測試在 DisposeAsync 刪 temp 資料夾時爆 `The process cannot access the file 'opencrawler.db'`。

**根因**:SQLite 開 WAL 模式後 `-wal` `-shm` journal 檔會被 pool 繼續持有,即便 Provider Dispose 了,Microsoft.Data.Sqlite 的連線 pool 還沒釋放檔案 handle。

**解法**:測試 DisposeAsync 呼叫 `SqliteConnection.ClearAllPools()` + `GC.Collect()` + `Task.Delay(100)` 再刪資料夾,且 `try/catch IOException` 寬容失敗。

```csharp
public async Task DisposeAsync()
{
    _server.Stop();
    Microsoft.Data.Sqlite.SqliteConnection.ClearAllPools();
    GC.Collect();
    GC.WaitForPendingFinalizers();
    await Task.Delay(100);
    if (Directory.Exists(_tempStorage))
        try { Directory.Delete(_tempStorage, true); }
        catch (IOException) { }
}
```

教訓:**Windows + SQLite WAL + 測試清理** 永遠記得 ClearAllPools。

### 坑 3. 資料夾名時間戳衝突

**症狀**:IT-02「同 URL 重複下載」時第二篇覆蓋第一篇,測試 `Assert.Equal(2, subdirs.Length)` 失敗。

**根因**:`GenerateArticleFolderName` 最初用 `yyyyMMddHHmmss` 精度,同一秒內連兩次下載結果同名。

**解法**:精度加到毫秒 `yyyyMMddHHmmssfff`,亂數 hash 從 2 bytes 加到 3 bytes。

教訓:**測試「時間導向的唯一鍵」必須跑連續 call 驗證**,時間精度永遠要比你以為需要的更高一級。

### 坑 4. ConfigService 測試污染全域

**症狀**:第一個測試跑完,全域 `%APPDATA%\openCrawler\config.json` 被改成指向測試暫存路徑,下次啟動 GUI 指錯路徑。

**根因**:CLI 的 `--storage` 參數透過 `cfg.SaveAsync(...)` 寫入磁碟,污染全域。

**解法**:加 `IConfigService.ApplyInMemory(cfg)`,CLI 下的 storage override 只改記憶體不寫檔。

教訓:**測試覆寫生產設定**是常見反模式;凡是 service 有 load/save 成對,要留一個純記憶體 setter。

### 坑 5. Avalonia Grid `ColumnSpacing` 不存在

**症狀**:build 丟 `AVLN2000: Unable to resolve suitable regular or attached property ColumnSpacing on type Grid`。

**根因**:`ColumnSpacing` 是 WPF 概念,Avalonia Grid 沒有這屬性。

**解法**:每個子元素上加 `Margin="0,0,8,0"`。或用 `StackPanel Orientation="Horizontal" Spacing="8"` 當替代。

教訓:**WPF → Avalonia 的屬性名不是 1:1 對應**,遇到就查,別靠猜。

### 坑 6. WebView.Avalonia 的三段式依賴

**症狀**:只裝 `WebView.Avalonia.Desktop` 時 `using AvaloniaWebView;` 找不到、`.UseDesktopWebView()` 也 unresolved。

**根因**:`WebView.Avalonia.Desktop` 這個 meta 只含平台包,**不自動**依賴核心 `WebView.Avalonia`(有 `WebView` 控制項本身與 `AvaloniaWebViewBuilder`)。

**解法**:csproj 同時 reference `WebView.Avalonia` + `WebView.Avalonia.Desktop`,並且:

```xml
<PackageReference Include="WebView.Avalonia" Version="11.0.0.1" />
<PackageReference Include="WebView.Avalonia.Desktop" Version="11.0.0.1" />
```

```csharp
using AvaloniaWebView;                 // WebView control + Initialize
using Avalonia.WebView.Desktop;        // UseDesktopWebView extension

AvaloniaWebViewBuilder.Initialize(default);
AppBuilder.Configure<App>().UsePlatformDetect().UseDesktopWebView()
```

教訓:**兩個命名空間** (`AvaloniaWebView` 與 `Avalonia.WebView.Desktop`) 不同,前者是控制項,後者是 AppBuilder 擴充。

### 坑 7. WebView 啟動炸「Unable to create child window for native control host」

**症狀**:App 啟動 → 例外 `Unable to create child window for native control host. Application manifest with supported OS list might be required.`

**根因**:Avalonia 的 `NativeControlHost`(WebView 基底)在 Windows 上需要 app manifest 宣告 `<compatibility>` + `<supportedOS>` GUIDs,否則底層 `CreateWindow` 呼叫拿不到必要的 Windows version API。

**解法**:在 `app.manifest` 加:

```xml
<compatibility xmlns="urn:schemas-microsoft-com:compatibility.v1">
  <application>
    <supportedOS Id="{e2011457-1546-43c5-a5fe-008deee3d3f0}"/>
    <supportedOS Id="{35138b9a-5d96-4fbd-8e2d-a2440225f93a}"/>
    <supportedOS Id="{4a2f28e3-53b9-4441-ba9c-d69d4a4a6e38}"/>
    <supportedOS Id="{1f676c76-80e1-4239-95bb-83d0f6d0da78}"/>
    <supportedOS Id="{8e0f7a12-bfb3-4fe8-b9a5-48fd50a15a9a}"/>
  </application>
</compatibility>
```

教訓:**WebView on Avalonia 在 Windows = 必備 compatibility manifest**,遇到 NativeControlHost 錯誤第一個看這個。

### 坑 8. 微信圖片全部變成同一張

**症狀**:用戶螢幕截圖顯示 20 張圖都是同一張「數字化星球聯盟 midoo 米多」banner。

**根因**:微信所有圖 URL 形如 `https://mmbiz.qpic.cn/mmbiz_jpg/xxxxx/640?wx_fmt=jpeg`,`Path.GetFileName(u.LocalPath)` 對所有圖都回 `640`。我原本的 `GenerateAssetFileName` 先試用 URL 的 filename(預期唯一)、沒 filename 才退回 hash。結果 WeChat 所有圖寫進 `assets/640`,後寫的蓋掉前寫的,最後只剩一張,HTML 裡所有 `<img>` 都指向同一個檔。

**解法**:**永遠用 URL 的 SHA1 12 碼當檔名核心**,副檔名從路徑或 Content-Type 推:

```csharp
var hash = Convert.ToHexString(SHA1.HashData(Encoding.UTF8.GetBytes(url)))[..12].ToLowerInvariant();
return $"asset_{hash}{ext}";
```

教訓:**命名唯一性絕不能依賴 URL 的 Path.GetFileName**,這是近十年越來越多「content-addressed-by-hash CDN」的反模式。永遠以 full URL hash 為唯一鍵。

### 坑 9. SPA 下載後預覽空白

**症狀**:微信、Medium 等網站下載後,本地開啟只有標題 + 作者,正文不見。

**根因一**:圖片在 HTML 裡是 `data-src`,真 `src` 為空(lazy load)。我們只抓 `src` → 沒抓到圖。

**根因二**:WeChat 正文在 `#js_content`,CSS 一開始設 `visibility: hidden` 或類似,要 JS 執行完才 un-hide。離線開啟時 JS 被剝,CSS 保留,內容永遠看不到。

**解法**:
1. CollectAssetUrls 多掃 `data-src / data-original / data-lazy-src / data-actualsrc / data-echo`。
2. RewriteResourceUrls 優先用 lazy 屬性值覆寫 `src`(這樣本地就直接顯示)。
3. 寫入前在 `<head>` 注入一個 offline visibility patch style,強制 `#js_content / .rich_media_content / body / main / article` 等常見容器 `visibility:visible;display:block;opacity:1;`。

教訓:**離線閱讀 = 內容 + 排版 + 必要時的 CSS hack**。只下載資源不夠,必須重現「JS 執行後」的可視化狀態。

### 坑 10. ChallengeDetector 沒觸發 browser fallback

**症狀**:明明是 SPA 網站,fast 模式跑完 `fetchMode: "fast"`,沒退到 browser。

**根因**:原本只看 `Html.Length < 500` 或有「Just a moment…」字串。WeChat 的靜態 HTML 有 100+ KB(所有 JS 片段)但可見 body 文字很短。

**解法**:在 ChallengeDetector 解析 DOM 後,檢查「body 可見文字 < 300 字 且 段落元素 < 3 且 `<img>` < 2」→ 視為需要 JS 渲染,回傳 true 觸發 browser fallback。

教訓:**只看檔長 ≠ 知道有沒有內容**,必須解析 DOM 看語意結構。

## 4. 規劃與實作的偏離點

| 規劃 | 實作 | 偏離原因 |
|------|------|---------|
| 主畫面三欄(分類 / 文章 / 詳情) | **兩欄:樹 + 詳情** | 使用者試用後反饋三欄切換動線長,改單一 TreeView 更自然 |
| v1 閱讀僅用系統瀏覽器(ADR-03) | **內嵌 WebView + 保留系統瀏覽器** | 使用者強烈要求內嵌,WebView.Avalonia 成熟可用,收益 > 複雜度 |
| ArticleNote 1:1 對應 | **1:N 多 note 分頁** | 使用者要求單篇可累積多個 note |
| Gemini 透過 Vertex AI | **預設 AI Studio API Key**,Vertex AI 保留選項 | API Key 對個人使用者入門門檻低太多 |
| Playwright binary 要求首次提示下載 | **偵測失敗才自動安裝**(lazy) | 實作簡單且使用者體感順暢 |
